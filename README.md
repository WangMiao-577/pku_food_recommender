# 北大食堂智能推荐系统 v2.0 — 设计文档

---

## Part 1: 视觉设计系统

### 概述

系统延续并扩展了原有水彩印象派UI风格，在保留温暖治愈的视觉基调之上，新增双模式切换的视觉标识体系。整体美学方向为"温馨柔和的数字化水彩"，以低饱和度的粉彩为主调，圆角卡片式布局贯穿全局，强调零压迫感的用餐决策体验。

- **离线模式**采用暖色调（珊瑚/暖金），传递稳定可靠的熟悉感
- **AI模式**采用冷色调（薄荷绿/薰衣草紫），传递智能探索的未来感
- 两种模式的所有功能页面输出结构完全统一，仅底层原理不同

## Part 2: 全局交互与动画

### 模式切换系统（核心全局交互）

**双模式切换逻辑**:
- 全局顶部导航栏右侧放置模式切换胶囊控件
- 离线模式（默认）: 所有推荐逻辑在本地运行，地理位置需用户手动输入
- AI模式: 所有推荐逻辑通过大模型API驱动，地理位置可自动获取（IP定位/设备定位）
- 切换时: 全页渐隐渐现（300ms），色彩主题从暖色系平滑过渡到冷色系
- 模式偏好保存在 localStorage，下次打开默认使用上次选择的模式

**模式切换动画**:
```
1. 用户点击模式切换胶囊
2. 当前页面内容 opacity:1→0, 150ms ease-out
3. 主题色变量CSS transition, 200ms
4. 新页面内容 opacity:0→1, 200ms ease-in
5. 模式标签（离线/AI）在胶囊内滑动切换, 300ms spring(1,80,10,0)
```

### 页面过渡动画

- **路由切换**: 新页面从右侧滑入（translateX: 30px→0），旧页面向左淡出（opacity:1→0, translateX:0→-20px），时长300ms，缓动函数 ease-out-cubic
- **卡片列表入场**: 瀑布流式 stagger 入场，每张卡片延迟50ms，从下方20px+opacity0滑入，时长250ms，spring缓动
- **套餐卡片特殊入场**: 横向展开（scaleX:0.9→1）+ opacity 0→1，时长350ms，ease-out，突出套餐的完整性

### 滚动行为

- 首页美食故事区域: 横向卡片轮播，支持 touch 滑动和鼠标拖拽， snapping 到卡片边界
- 相关菜品列表: 横向滚动容器，隐藏滚动条，右侧渐隐遮罩提示更多内容
- 返回顶部: 浮动圆形按钮（右下，48px），滚动超过500px后淡入，点击平滑滚动到顶部（500ms ease-in-out）

### 通用微交互

**按钮交互**:
- Hover: 亮度提升10%（filter: brightness(1.1)），100ms ease
- Active/Press: scale(0.97)，50ms ease
- 幽灵按钮（如"找店指引"）: hover时背景填充主色，文字变白，150ms

**标签交互**:
- 口味/场景标签: hover时背景加深，cursor pointer
- 选中态: 实心主色背景+白字

**输入框交互**:
- Focus: 边框变为主色，外发光 0 0 0 3px 主色20%透明度，150ms
- AI模式下对话输入: 底部固定，发送按钮hover时旋转15度

**加载状态**:
- 通用骨架屏: 渐变动画（shimmer），基色 `#F0EAE3`，高光 `#F5F0EA`，1.5s循环
- AI思考状态: 顶部薄荷绿色进度条（indeterminate）+ 界面半透明遮罩
- 找店指引地图加载: 脉冲动画的中心定位图标

**数字动画**:
- 营养值/评分变化: 数字滚动计数（500ms ease-out）
- 价格显示: 无动画，直接渲染

**手势支持**:
- 移动端首页故事卡片: 左右滑动手势切换
- 菜品详情页: 左滑返回上一页
- 找店指引浮层: 下滑关闭（移动端）

### Section: 稳定模式 vs 探索模式推荐

**稳定模式行为**:

- 探索增强: ε-greedy探索概率提升至40%
- 视觉标识: 菜品卡片右上角紫色pill "新发现" / "探索推荐"

### Section: 推荐系统算法架构（召回+排序两阶段）

**召回阶段（Retrieve）**:

多路召回策略，三路并行：

1. **场景召回**: 根据用餐场景筛选
   - 独自速食 → prep_time≤8min + 价格适中
   - 独自慢食 → 评分≥4.0 + 特色菜品
   - 同伴聚餐 → 大份量/套餐组合 + 热门菜品
   - 团体宴请 → 支持多人份/高端菜品 + 大食堂

2. **需求召回**: 根据用户输入的具体需求筛选
   - 预算硬约束: price ≤ budget_limit
   - 口味软约束: flavor与偏好匹配（至少命中1个）
   - 营养约束: 热量/蛋白质/脂肪在目标范围内
   - 地理位置约束: 距离用户当前位置≤设定阈值
   - 时段约束: 当前时间在供应时段内

3. **模式召回**: 根据稳定/探索模式筛选
   - 稳定模式: 历史消费菜品池（近30天吃过的菜）+ 热门兜底
   - 探索模式: 全量菜品池 - 近7天已吃 - 明确不喜欢

**排序阶段（Rank）**:

多因子加权排序公式:

```
final_score = α * preference_score + β * quality_score + γ * location_score + δ * freshness_score + ε * explore_score

其中:
- preference_score: 用户画像匹配度（口味/营养/价格偏好），0-100
- quality_score: 菜品质量分（评分+评价人数+卖相），0-100
- location_score: 距离衰减分（越近越高），0-100
- freshness_score: 新鲜度分（多久没吃），0-100
- explore_score: 探索 bonus（新菜品加分），0-30

权重分配:
- 稳定模式: α=0.35, β=0.25, γ=0.20, δ=0.15, ε=0.05
- 探索模式: α=0.25, β=0.20, γ=0.15, δ=0.10, ε=0.30
```

**冷启动处理**:

1. **全新用户**（无历史+无画像）:
   - 首次进入: 弹出3个快速问题（口味偏好/预算/营养目标）
   - 使用热门菜品 + 口味匹配召回
   - 排序以质量分为主（β=0.5）
   
2. **有画像无历史**:
   - 直接使用画像偏好进行召回排序
   
3. **有历史无画像**:
   - 从历史记录反向推导画像（最常吃的口味/食堂/价格区间）

4. **用户画像持久化**:
   - 存储在 `user_profile.json` 中
   - 每次推荐后更新（强化学习式反馈: 点击+1分，就餐+2分，跳过-0.5分）

## Part 7: 技术栈

- **前端**: PyQt5（延续原有技术栈）
- **数据存储**: JSON文件（延续原有结构，扩展字段）
- **AI模式**: 大模型API（OpenAI/Claude/文心一言，通过 `requests` 库调用）
- **定位服务**: AI模式下使用IP定位（`requests` 获取IP地理位置）
- **图表**: `matplotlib` 或 `pyqtgraph`（营养可视化）
- **可选依赖**: `requests`（API调用）, `matplotlib`（数据可视化）
